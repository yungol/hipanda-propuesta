Push a la rama felipe y el código aparece del otro lado. Sin pasos manuales.

Se configuró un pipeline CI/CD que, ante cada push a la rama felipe en los repositorios de producción, replica automáticamente el código completo a los repositorios de Felipe en GitLab. Tres front-ends y dos back-ends sincronizados sin intervención humana.

1. Arquitectura de la solución

Un job de CI que clona y replica. Simple en concepto, con bordes que hubo que resolver.

Cuando se hace push a la rama felipe, GitLab CI ejecuta un job que obtiene el código completo y lo empuja al repositorio destino usando un token de acceso personal. Sin servidores intermedios, sin cron jobs externos, sin sincronización manual.

3. Lo que hubo que resolver

Cinco obstáculos técnicos que atraviesan la superficie de «un simple git push».

Cada uno de estos problemas bloqueó el pipeline en algún momento. Resolverlos fue lo que convirtió la idea en algo que efectivamente corre. Entenderlos ayuda a ver por qué la solución final tiene la forma que tiene.

1

SSH en Alpine no es SSH en macOS

La intuición natural es usar llaves SSH para la autenticación entre repositorios. Pero Alpine Linux (la imagen que corre el pipeline, apenas 20 MB) usa OpenSSH con una implementación que rechaza las llaves generadas con el ssh-keygen de macOS. El pipeline fallaba antes de empezar, por una incompatibilidad de formato que no da ninguna pista útil en los logs.

Resuelto HTTPS con Personal Access Token: autenticación que funciona en cualquier entorno sin depender del sistema operativo del runner.
2

Deploy Tokens no tienen write_repository

GitLab ofrece Deploy Tokens para operaciones automatizadas. El problema: su scope más alto es read_repository. Para hacer un push necesitás write_repository, y ese permiso solo lo otorgan los Personal Access Tokens. Parece un detalle de permisos, pero si elegís la herramienta incorrecta al inicio, perdés tiempo antes de darte cuenta de que el scope simplemente no existe.

Resuelto Personal Access Token con scope write_repository configurado como variable CI.
3

Variables protegidas no llegan a ramas no protegidas

GitLab tiene una opción de «Protected» en las variables CI. Activarla es buena práctica de seguridad. Pero las variables protegidas solo se inyectan en ramas protegidas (típicamente main o master). La rama felipe no está protegida, así que el token nunca llegaba al job y la autenticación fallaba sin un mensaje claro. El comportamiento es correcto según la documentación, pero contraintuitivo si no conocés el modelo de seguridad de GitLab CI.

Resuelto Variable FELIPE_TOKEN sin el flag Protected, manteniendo el flag Masked activo.
4

Shallow clone: el runner no tiene la historia completa

Por defecto, GitLab Runner clona los repositorios con profundidad limitada para ahorrar tiempo y espacio. El resultado es un clone superficial (shallow) que GitLab.com rechaza al intentar hacer push porque le faltan objetos en la historia. El error es genérico y no sugiere la causa. La solución existe pero no es obvia sin conocer la arquitectura de clones de GitLab CI.

Resuelto git fetch --unshallow al inicio del job para obtener la historia completa antes del push.
5

Refspec ambiguo en el primer push a un repo vacío

Cuando el repositorio destino todavía no tiene la rama felipe, Git no puede resolver una referencia ambigua con la sintaxis abreviada. El push falla con un mensaje críptico sobre refspecs. La solución técnica es simple, pero diagnosticarlo requiere entender el protocolo interno de Git para resolución de referencias en repositorios remotos sin esa rama creada previamente.

Resuelto HEAD:refs/heads/felipe en lugar de la sintaxis abreviada para el primer push.

4. Operación y seguridad

Una vez configurado, el flujo diario es de dos comandos.

Flujo de trabajo diario

$ git checkout felipe
$ git merge master          # traer lo último de producción
$ git push origin felipe    # el CI hace el resto

Lo que no hay que hacer nunca más

  • - Clonar manualmente el repo a una carpeta local de Felipe
  • - Copiar archivos entre máquinas o por correo
  • - Mantener dos remotes locales y pushear dos veces
  • - Sincronizar ramas a mano después de cada cambio

Consideraciones de seguridad

  • ✓ Token almacenado como variable CI enmascarada
  • ✓ Autenticación sobre HTTPS, sin llaves SSH en el runner
  • ✓ Scope del token limitado a write_repository
  • ✓ Se recomienda rotar el PAT periódicamente

5. Instrucciones para Felipe

Lo que tenés que hacer en tu equipo local para levantar cada proyecto.

El pipeline ya te dejó el código actualizado en GitLab en la rama felipe. Desde ahí vos decidís cómo correrlo. Esto aplica a los cinco repositorios.

1

Hacer pull del repo y pararse en la rama felipe

La rama felipe es la que tiene la última versión sincronizada. Siempre empezá desde acá.

En tu terminal

$ git clone git@gitlab.com:galco2/[repo].git
$ cd [repo]
$ git checkout felipe
$ git pull origin felipe
2

Antes de arrancar: configurar las variables de entorno

Cada proyecto necesita su archivo .env con las variables de entorno. Sin esto los back-ends no conectan a la base de datos y los front-ends no saben a qué API pegarle.

⚠️ No omitir este paso

Los .env no se suben al repo por seguridad (están en .gitignore). Pedile a Juan las variables de entorno de cada proyecto o copialas de tu backup. Ponelas en un archivo .env en la raíz de cada repo antes de ejecutar cualquier comando.

3

Tres formas de levantar el proyecto

Tenés tres caminos según lo que necesites. Los tres parten de tener el repo clonado y el .env configurado.

Opción A — Correr directo en la rama felipe

La más rápida. El código que llegó por el pipeline ya está listo. Ideal para probar cambios rápido.

$ git checkout felipe
$ npm install
$ npm run dev

Opción B — Mergear felipe en dev y correr en dev

Si querés integrar lo de felipe en la rama de desarrollo antes de pasarlo a producción.

$ git checkout dev
$ git merge felipe
$ npm install
$ npm run dev

Opción C — Mergear felipe en master y correr en master

Cuando el cambio ya está probado y querés pasarlo a la rama principal de producción local.

$ git checkout master
$ git merge felipe
$ npm install
$ npm run dev

Checklist por repositorio

1 Clonar el repo de galco2 en tu máquina
2 git checkout felipe para tener la última versión
3 Crear el .env con las variables de entorno
4 Elegir opción A, B o C según la rama donde quieras correr
5 npm install && npm run dev

✅ Resumen

El pipeline te ahorra el trabajo de sincronizar código. De ahí en adelante es estándar: pull, .env, checkout a la rama que quieras, npm install, npm run dev. Repetí el checklist en cada uno de los 5 repositorios y listo.

Resumen

El pipeline elimina la fricción de mantener dos copias del código sincronizadas.

La ventaja no es de complejidad técnica exhibida, sino de complejidad operativa eliminada. Sin el pipeline, cada sincronización entre los repositorios de producción y los de Felipe requiere pasos manuales, propensos a error y a desactualización. Con el pipeline, el push a la rama felipe es el único paso consciente; la replicación ocurre sola, con trazabilidad completa en el log de CI y sin depender de que alguien recuerde hacerlo.
